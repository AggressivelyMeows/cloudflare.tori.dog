**D1** is Cloudflare's serverless SQL database. You write standard SQL queries, and D1 stores the data and runs them on Cloudflare's network.

> **Think of it as:** a SQLite database that lives on Cloudflare's edge, with no servers for you to manage.

## The 4 Words You'll See Most

**Database**
A named container for your tables and data. You might have one database for an app's users and another for analytics. In your `wrangler.jsonc`, you connect a database to your Worker as a binding.

**Binding**
The name your Worker uses to reach a database. If your binding is `MY_DB`, you query it as `env.MY_DB.prepare(...)`.

**Migration**
A versioned SQL file that changes the schema. Wrangler tracks which migrations have run, so each database only gets each change once.

**Query**
A SQL statement run through `prepare` and `bind`. `prepare` returns a statement you can parameterise with `bind`, then execute with `run`, `first`, or `all`.

## A Tiny Example

Here's how you might insert and read a row inside a Worker:

```js
export default {
  async fetch(request, env) {
    // insert a row
    await env.MY_DB.prepare("INSERT INTO links (slug, url) VALUES (?, ?)")
      .bind("tor", "https://example.com")
      .run();

    // read it back
    const row = await env.MY_DB.prepare("SELECT url FROM links WHERE slug = ?")
      .bind("tor")
      .first();

    return new Response(row.url);
  }
};
```

`MY_DB` is the binding name you set in `wrangler.jsonc`. The `?` placeholders are filled in by `bind`, in order.

## When Would I Use D1?

D1 fits when your data is **relational**: rows that reference each other, queries that join or filter, and schemas that change over time. Use it for user accounts, product catalogs, content tables, or anything where SQL is the natural shape of the question.

For a single key looked up by name, <a class="tori-glossary-link" href="?glossary=kv">KV</a> is simpler and faster. For large binary files, use <a class="tori-glossary-link" href="?glossary=r2">R2</a>.
